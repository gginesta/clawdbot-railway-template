## Leonardo Anthropic Auth Debugging (Day-long incident)

### Problem
Leonardo's Anthropic auth has been broken since initial setup. Token in auth store shows as 12 chars (`sk-ant-oat01` — just the prefix) or 0 chars (empty) instead of full 108-char OAuth token.

### Root Cause Found
Two auth files exist in OpenClaw:
- `/data/.openclaw/agents/main/agent/auth.json` — **TRUE source** (gateway reads this on startup)
- `/data/.openclaw/agents/main/agent/auth-profiles.json` — derived/cache (generated FROM auth.json by gateway)

All previous fix attempts wrote to `auth-profiles.json` — the wrong file. This is why nothing persisted.

### auth.json structure (from Molty's working container):
```json
{
  "anthropic": {"type": "api_key", "key": "sk-ant-oat01-..."},
  "openai-codex": {"type": "oauth", "access": "...", "refresh": "...", "expires": ...},
  ...
}
```

### Fix Attempts (all failed remotely):
1. Webhooks writing to auth-profiles.json → wrong file
2. Added ANTHROPIC_API_KEY env var (wrong — OAuth tokens can't use x-api-key header) → deleted
3. Webhooks writing to auth.json → env var (`ANTHROPIC_TOKEN`) not available in isolated sub-agent processes, wrote empty string
4. Gateway config.patch via sub-agent → gateway tool couldn't set token value
5. Updated Railway service start command to fix auth.json before startup → "Auth fixed: 108 chars" in logs BUT gateway resets auth.json to empty during its own startup sequence
6. Hardcoded token in Python script → script ran but gateway overwrote

### Why Gateway Resets auth.json:
The OpenClaw gateway initializes its in-memory auth state and WRITES it to auth.json during startup. This overwrites any file-level changes made before the gateway writes. The gateway's in-memory state has the truncated/empty token from the original botched setup. Writing to auth.json only works if done through the gateway's own auth API (paste-token command).

## Key Lessons Learned Today

- **auth.json vs auth-profiles.json**: auth.json is the TRUE source. auth-profiles.json is generated BY the gateway. Never write to auth-profiles.json directly.
- **Isolated sub-agent processes do NOT have container env vars**: ANTHROPIC_TOKEN (and similar) are not available in isolated webhook sub-agents. Scripts relying on env vars will get empty strings.
- **Railway CLI `railway shell` = LOCAL subshell with env vars, NOT container shell**. `railway connect` is needed to SSH into the actual container.
- **paste-token is the right tool** for Anthropic OAuth token setup — it updates through the gateway's auth API, which persists correctly. Interactive TTY required.
- **Webhooks are unreliable for file system operations** — sub-agents spawn in ephemeral contexts without full container env access.
