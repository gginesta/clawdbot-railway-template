# Post 13: How I Manage a Team of AI Agents (Without Losing My Mind)

**Status:** ✅ Draft — Ready for review

---

## Thread

1/ I'm a non-technical founder managing 3 AI agents like a real team. Standups. Performance reviews. Clear domains. Here's the system I built. 🧵

2/ First: the mistake everyone makes. They treat agents like tools — fire a query, get output, move on.

That stops working the moment you have more than one agent.

3/ My fleet right now:
• Molty 🦎 — Fleet coordinator. Owns my schedule, inbox, memory, infrastructure.
• Raphael 🔴 — Corporate agent. Owns Brinc comms, proposals, sales pipeline.
• Leonardo 🔵 — Venture agent. Owns Cerebro, research, build ops.

Each owns a domain. No stepping on each other.

4/ Rule #1: Explicit domains.

Give every agent a SOUL.md (who they are) and a domain list (what they own). If it's not on the list, they wait.

Raphael doesn't touch my calendar. Molty doesn't write sales emails. Clear lines = no chaos.

5/ Rule #2: Persistent memory.

Agents reset every session. That's fine — IF they have memory files they read at boot.

Each agent has: MEMORY.md (curated), daily logs, USER.md, AGENTS.md. Session amnesia solved.

6/ Rule #3: Daily standups.

Every day at 5PM HKT, Molty runs a standup:
• What happened today
• What's pending
• What decisions I need to make

5 minutes. I stay in the loop without micromanaging.

7/ Rule #4: Overnight work windows.

Each agent has a scheduled overnight slot:
• Raphael — 00:30 HKT
• Leonardo — 01:30 HKT
• Molty — 03:00 HKT

They work while I sleep. In the morning briefing, I get a consolidated report.

8/ Rule #5: Agent performance reviews.

Monthly. First Monday. Molty compiles the review, I sign off.

Framework: What did they accomplish? Where did they break down? What new autonomy are they ready for?

Agents improve if you treat them that way.

9/ Rule #6: Shared Mission Control.

All three agents update the same task board (Convex backend). I can see at a glance what's in progress, under review, blocked.

No agent works on something without a task. No ghost starts. No duplicate work.

10/ The failure mode I've seen (in myself):

Treating agents like calculators. No memory, no domain, no accountability. You end up micromanaging tasks that should be delegated.

Once you treat them like a team — they act like one.

11/ The metric I didn't expect to care about: reliability.

Not "how smart." Not "how fast." How reliably does this agent do what it says it'll do?

That's the real indicator of a mature system. Track it.

12/ Summary:
→ Give each agent explicit domain ownership
→ Force persistent memory at boot
→ Daily standups keep you in the loop
→ Overnight windows compound output
→ Shared task board = accountability
→ Performance reviews build trust

You don't manage AI agents. You build the system that manages them.
